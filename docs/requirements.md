# 企业科技管理平台 — 需求规格说明书

## 1. 项目概述

### 1.1 项目名称

企业科技管理平台（Enterprise Tech Management Platform）

### 1.2 项目目标

建设一个面向企业内部的科技管理 Web 应用，提供用户管理、权限控制等通用能力，并支撑清单管理、奖惩记录、科技问题反馈三大业务模块，提升日常工作效率和规范化水平。

---

## 2. 用户与角色

| 角色 | 说明 |
|------|------|
| **普通用户** | 默认角色，可使用清单管理、查询奖惩记录、提交科技问题反馈 |
| **部门文书** | 每个部门一名，继承普通用户权限，额外拥有本部门奖惩记录的增删改查权限 |
| **问题管理员** | 继承普通用户权限，额外负责问题分类、指派整改责任人 |
| **系统管理员** | 最高权限，管理用户、分配角色、系统配置 |

### 2.1 权限矩阵

| 功能 | 普通用户 | 部门文书 | 问题管理员 | 系统管理员 |
|------|:--:|:--:|:--:|:--:|
| 个人清单 CRUD | ● | ● | ● | ● |
| 查询奖惩记录 | ● | ● | ● | ● |
| 管理部门奖惩记录 | ○ | ● | ○ | ○ |
| 提交问题反馈 | ● | ● | ● | ● |
| 问题分类/指派 | ○ | ○ | ● | ○ |
| 填写整改方案/时限 | ○ | ○ | ●（被指派为责任人时） |
| 确认问题完成 | ●（仅自己的问题） | ● | ● | ● |
| 用户管理 | ○ | ○ | ○ | ● |
| 角色分配 | ○ | ○ | ○ | ● |

- ● 有权限 ○ 无权限
- ※ 查询奖惩记录：仅限查看本部门的记录

## 3. 功能模块

### 3.1 用户与权限管理

**3.1.1 用户管理**
- 用户注册/登录（账号+密码）
- 用户基本信息：姓名、部门、职位、联系方式
- 账号启用/禁用

**3.1.2 角色管理**
- 系统预设四种角色，管理员可为用户分配角色
- 一个用户可拥有多个角色（权限叠加）

---

### 3.2 清单管理

**目标用户**：所有用户  
**核心目的**：记录领导交办事项、重要待办，避免遗漏

**3.2.1 功能列表**

| 功能 | 说明 |
|------|------|
| 新建清单 | 创建一条工作待办事项 |
| 编辑清单 | 修改自己创建的清单 |
| 删除清单 | 删除自己创建的清单（软删除） |
| 查询清单 | 按状态、时间范围、关键词筛选 |
| 状态流转 | 待办 → 进行中 → 已完成 |

**3.2.2 数据字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| 事件来源 | 下拉选项 | ● | 领导交办/会议部署/自行安排/其他 |
| 事项描述 | 文本域 | ● | 待办事项的具体内容 |
| 计划完成时间 | 日期 | ○ | 预计完成的日期 |
| 实际完成时间 | 日期 | ○ | 实际完成日期，状态为"已完成"时填写 |
| 事项状态 | 下拉选项 | ● | 待办/进行中/已完成 |
| 责任人 | 文本 | ● | 默认当前用户，可修改 |
| 备注 | 文本域 | ○ | 补充说明 |
| 创建时间 | 日期时间 | 自动 | |
| 更新时间 | 日期时间 | 自动 | |

**3.2.3 业务规则**
- 用户只能查看和操作自己创建的清单
- 删除为软删除，不在列表中展示但保留数据
- 状态从"待办"→"进行中"→"已完成"单向流转（也可从"待办"直接→"已完成"）

---

### 3.3 奖惩记录

**目标用户**：部门文书（管理）、本部门人员（查看）  
**核心目的**：记录部门内的奖励与惩罚事项，仅本部门内可见

**3.3.1 功能列表**

| 功能 | 普通用户 | 部门文书 |
|------|:--:|:--:|
| 新增记录 | ○ | ● |
| 编辑记录 | ○ | ● |
| 删除记录 | ○ | ●（本部门） |
| 查询/查看 | ● | ● |
| 导出列表 | ○ | ● |

**3.3.2 数据字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| 类型 | 下拉选项 | ● | 奖励/惩罚 |
| 标题 | 文本 | ● | 简要标题 |
| 内容描述 | 文本域 | ● | 奖惩事由详情 |
| 涉及人员 | 文本 | ● | 被奖励/惩罚的人员姓名 |
| 涉及部门 | 下拉选项 | ● | 所属部门 |
| 决定日期 | 日期 | ● | 奖惩决定的日期 |
| 文号 | 文本 | ○ | 相关文件编号 |
| 附件 | 文件上传 | ○ | 相关证明材料 |
| 创建人 | 自动 | ● | 部门文书 |
| 创建时间 | 日期时间 | 自动 | |
| 更新时间 | 日期时间 | 自动 | |

**3.3.3 业务规则**
- 每个部门只有一名部门文书，可以增删改本部门记录
- 奖惩记录仅本部门内可见，不可跨部门查看
- 系统管理员不受部门限制
- 部门内人员均可查询，支持按类型、时间范围筛选
- 删除为软删除

### 3.4 科技问题反馈

**目标用户**：所有人（提交）、问题管理员（审批/指派）、整改责任人（落实）  
**核心目的**：收集各部门对信息科技的吐槽和问题，形成整改闭环

**3.4.1 审批流**

```
问题提出人                    问题管理员                   整改责任人                  问题提出人
  │                              │                           │                          │
  │  ① 提交问题                   │                           │                          │
  │ ──────────────────────────▶  │                           │                          │
  │                              │  ② 判断问题类型             │                          │
  │                              │     指派整改责任人            │                          │
  │                              │ ────────────────────────▶  │                          │
  │                              │                           │  ③ 填写整改方案           │
  │                              │                           │     填写整改完成时限        │
  │                              │                           │ ───▶ 状态：整改中          │
  │                              │                           │                          │
  │                              │                           │  ④ 整改完成              │
  │                              │                           │ ───▶ 状态：待确认          │
  │                              │                           │                          │
  │  ⑤ 确认完成                    │                           │                          │
  │ ───▶ 状态：已完成              │                           │                          │
  │                              │                           │                          │
  │  ⑥ 不满意，退回重改              │                           │                          │
  │ ───▶ 状态：整改中              │                           │                          │
```

**3.4.2 状态说明**

| 状态 | 含义 | 当前处理人 |
|------|------|-----|
| 待分派 | 已提交，等待问题管理员处理 | 问题管理员 |
| 整改中 | 已指派，责任人正在制定方案或整改 | 整改责任人 |
| 待确认 | 整改完成，等待提出人确认 | 问题提出人 |
| 已完成 | 提出人确认，问题关闭 | — |
| 已驳回 | 问题管理员驳回（不符合问题范围） | — |

**3.4.3 数据字段**

| 字段 | 类型 | 必填 | 阶段 | 说明 |
|------|------|:--:|------|------|
| 问题标题 | 文本 | ● | ① | 简要概述 |
| 问题描述 | 文本域 | ● | ① | 详细描述问题现象 |
| 反馈部门 | 下拉选项 | ● | ① | 问题提出人所属部门 |
| 问题分类 | 下拉选项 | ○ | ② | 由系统管理员在后台灵活配置分类项 |
| 驳回理由 | 文本域 | ○ | ② | 问题管理员驳回时填写 |
| 紧急程度 | 下拉选项 | ● | ① | 普通/紧急/非常紧急 |
| 整改责任人 | 用户选择 | ○ | ② | 问题管理员指派 |
| 整改措施 | 文本域 | ○ | ③ | 整改责任人填写 |
| 整改完成时限 | 日期 | ○ | ③ | 整改责任人填写 |
| 实际完成时间 | 日期时间 | 自动 | ④ | 责任人提交完成时记录 |
| 状态 | 状态枚举 | ● | — | 见上表 |
| 提出人 | 自动 | ● | ① | 当前用户 |
| 提出时间 | 日期时间 | 自动 | ① | |
| 更新时间 | 日期时间 | 自动 | — | |

**3.4.4 业务规则**
- 所有登录用户均可提交问题反馈
- 问题管理员收到待分派问题后，判断问题类型、指派整改责任人
- 问题管理员也可直接驳回（状态→已驳回），并填写驳回理由
- 整改责任人填写方案和时限后，状态自动变为"整改中"
- 整改完成时，整改责任人提交，状态变为"待确认"
- 问题提出人确认完成→状态变"已完成"；若不满意→状态退回"整改中"
- 每个状态变更记录操作日志（谁、何时、做了什么操作）

```
users（用户表）
  ── id, username, password, name, department, position, phone, enabled, created_at

## 6. 已确认事项

1. 部门文书：每个部门一名
2. 奖惩记录：仅本部门内可见
3. 问题分类：由系统管理员在后台灵活配置，预设默认分类（硬件/软件/网络/安全/数据/其他）
4. 消息通知：暂不需要
5. 数据统计看板：需要（各模块的数量统计、趋势图等）
6. 项目经理角色：暂不增加

---

## 7. 问题分类配置

系统管理员可在后台管理问题分类，包括：
- 查看分类列表
- 新增分类（名称、排序）
- 编辑分类
- 启用/禁用分类（禁用的分类在新建问题时不可选）
- 预设默认分类：硬件、软件、网络、安全、数据、其他

对应数据表：
```
issue_categories（问题分类表）
  ── id, name, sort_order, enabled, created_at
```
